# Аналіз: Лабораторна 2 — Шарова архітектура та доменна модель

## Що змінилося порівняно з лабораторною 1?

### Структура проєкту

У lab1 структура була плоскою: `models.py`, `schemas.py`, `routers/` — все в одній папці `app/`. Бізнес-логіка (перевірка наявності на складі, унікальність ISBN, заборона видалення книги з замовленнями) жила безпосередньо в роутерах поруч із SQL-запитами та HTTP-кодами.

У lab2 проєкт розділений на 4 шари, кожен — окремий пакет:

```
app/
├── domain/           # Бізнес-логіка: моделі, фабрики, інтерфейси репозиторіїв
├── application/      # Оркестрація: use cases
├── infrastructure/   # Технічна реалізація: ORM, БД, репозиторії
└── presentation/     # HTTP: контролери, DTO, exception handlers
```

Файлів стало значно більше (з ~8 до ~30), але кожен файл має одну відповідальність. Навігація ускладнилась, але `grep` та IDE-пошук повністю компенсують це.

### Ключові зміни

**Доменні моделі відокремлені від ORM.** У lab1 клас `Book` в `models.py` був одночасно і таблицею БД, і "бізнес-об'єктом". У lab2 є два різних класи: `Book` (domain/models/book.py) — чистий Python без залежностей від SQLAlchemy, та `BookEntity` (infrastructure/orm_models.py) — опис таблиці. `BookMapper` перекладає між ними.

**Value Objects замість примітивів.** ISBN перевіряє формат при створенні, Money не дозволяє від'ємні суми, Email нормалізує регістр. У lab1 ці перевірки були розкидані по Pydantic-схемах та контролерах. Тепер правило "якщо об'єкт існує — він валідний" гарантується самим Value Object.

**Rich Domain Model.** Метод `book.sell(3)` сам перевіряє, чи достатньо книг на складі, і кидає `DomainError` якщо ні. У lab1 ця логіка була в `routers/orders.py` як `if book.quantity < item.quantity: raise HTTPException(409, ...)`. Тепер модель захищає свій стан — зовнішній код не може привести `Book` до невалідного стану.

**Domain Factory** перевіряє складні інваріанти (унікальність ISBN) через інтерфейс репозиторію, не знаючи про конкретну БД. У lab1 ця перевірка була прямо в роутері з SQLAlchemy-запитом.

**Контролери стали тонкими.** Єдина задача роутера — маппінг HTTP → DTO → Use Case → Response DTO. Жодної бізнес-логіки.

## Переваги розділення на шари

**Тестуємість домену.** 30+ unit-тестів домену запускаються за мілісекунди без БД та фреймворку. У lab1 це було неможливо — бізнес-логіка жила в `async` функціях роутерів, які потребували `AsyncSession`.

**Ізоляція змін.** Зміна формату HTTP-відповіді зачіпає лише `presentation/dto/`. Зміна бізнес-правила — лише `domain/`. Зміна ORM — лише `infrastructure/`. У lab1 зміна в одному місці каскадно зачіпала інші.

**Зрозумілість бізнес-правил.** Відкривши `domain/models/book.py`, одразу видно всі правила: `sell()`, `restock()`, `is_low_stock()`, `update_info()`. У lab1 ці правила були розмазані по 3 різних роутерах.

## Недоліки / ускладнення

**Більше коду.** Кількість файлів зросла втричі. Для кожної сутності є: доменна модель, ORM-модель, mapper, repository interface, repository implementation, DTO request, DTO response, use case. Для проєкту з 3 ендпоінтами — це overkill.

**Маппінг.** `BookMapper.to_domain()` / `to_entity()` — це бойлерплейт. Але це ціна за ізоляцію: без нього зміна в ORM-моделі зламає доменну логіку.

**Складніша навігація.** Щоб зрозуміти, як працює створення книги, потрібно пройти: router → use case → factory → domain model → repository → mapper → ORM. У lab1 — один файл `routers/books.py`.

## Наскільки простіше замінити БД або фреймворк?

**БД.** Замінити PostgreSQL на MongoDB — це написати `MongoBookRepository`, який реалізує той самий інтерфейс `BookRepository`. Домен, use cases та presentation — без змін. У lab1 потрібно було переписати кожен роутер.

**Фреймворк.** Замінити FastAPI на Flask — переписати лише `presentation/`. Домен та use cases не зміняться. У lab1 — бізнес-логіка була переплетена з `Depends(get_db)`, `HTTPException`, `select()`.

## Чому обрано Rich Domain Model?

Обрано **Rich Domain Model**, бо SmartBook Inventory має бізнес-правила з переходами стану (sell/restock змінюють quantity з перевірками, order має lifecycle pending → completed/cancelled). Rich Model інкапсулює ці правила: `booking.sell(5)` замість `service.sell(booking, 5)`. Це запобігає дублюванню логіки, якщо в майбутньому з'являться інші сценарії списання книг.

Anemic Model була б виправдана для простішого CRUD без складних інваріантів. Але ключовий інваріант "не продати більше ніж є" — це саме та логіка, яка має жити всередині агрегату, а не в зовнішньому сервісі.

## ADR: Rich Domain Model

**Рішення:** Rich Domain Model  
**Обґрунтування:** Моделі системи мають нетривіальну поведінку: Book контролює кількість на складі через sell()/restock(), Order має lifecycle зі статусами. Приватні поля з properties захищають інваріанти — зовнішній код не може встановити quantity напряму, минаючи перевірку.  
**Альтернатива:** Anemic Model з BookService. Відхилено, бо правила sell/restock ризикують дублюватися при появі нових use cases.
