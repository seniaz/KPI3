# Аналіз лабораторної 4: Синхронна та асинхронна комунікація

## Що змінилося порівняно з лабораторною 3

У lab3 вся логіка виконувалась "в лінію": Controller → Command/QueryHandler → Domain → Repository. Побічних ефектів не було — handler створював замовлення, зменшував кількість книг і зберігав результат. Жодних нотифікацій, аудиту чи зовнішніх інтеграцій.

У lab4 з'явились три нові компоненти: допоміжний модуль `notification/` (SupplierNotificationService), механізм доставки подій (InMemoryEventBus в `infrastructure/event_bus/`), та доменні події (`domain/events/`). Ці компоненти дозволяють системі реагувати на бізнес-дії без збільшення зв'язаності основних handlers.

## Які побічні операції виділено і чому

Для домену книгарні найприроднішими побічними операціями є:

**1. Сповіщення постачальника про низький запас (Restocking Request).** Коли кількість примірників книги падає нижче 3 після продажу, система має сповістити постачальника. Це класична побічна операція — основна бізнес-дія (продаж) не повинна залежати від доступності email-сервісу чи API постачальника.

**2. Підтвердження замовлення (Order Confirmation).** Після успішного оформлення замовлення клієнт має отримати підтвердження. Знову ж — замовлення вже створене, підтвердження може зачекати секунду.

Обидві операції делеговані окремому модулю `notification/` з чітким контрактом (`SupplierNotificationService` — Protocol). Реалізація `LoggingSupplierNotificationService` логує замість реальної відправки, але контракт однаковий для production-реалізації.

## Порівняння синхронного та асинхронного підходів

### Час відповіді API

**Синхронний (`POST /orders/sync`):** Handler послідовно виконує: створення замовлення → збереження → відправка підтвердження → перевірка запасів → відправка restocking-запиту. Загальний час = сума всіх операцій. Якщо нотифікація займає 200мс, клієнт чекає на 200мс більше.

**Асинхронний (`POST /orders/async`):** Handler виконує: створення замовлення → збереження → публікація подій. Клієнт отримує відповідь одразу після збереження. Підписники обробляють події незалежно. Час відповіді = лише основна операція.

### Поведінка при збоях

**Синхронний:** Якщо `send_order_confirmation()` падає — весь запит повертає 500. Замовлення не створюється (транзакція відкочується). Для restocking-запитів реалізовано try-catch (м'яка деградація), але для підтвердження збій критичний. Це свідомий дизайн-вибір — підтвердження вважається обов'язковим.

**Асинхронний:** Якщо підписник падає — EventBus ловить виняток і логує помилку. Замовлення вже збережене. Основна операція ніколи не залежить від підписників. Це головна перевага — відмовостійкість.

### Зв'язаність між компонентами

**Синхронний:** `SyncCreateOrderCommandHandler` має залежність від `SupplierNotificationService` через конструктор. Handler знає, що існує нотифікаційний сервіс. Додати новий побічний ефект (аналітика, аудит) = додати нову залежність в handler.

**Асинхронний:** `AsyncCreateOrderCommandHandler` знає лише про `EventBus`. Він публікує факт `OrderPlaced` — хто на це реагує, не його справа. Додати нового підписника = додати `event_bus.subscribe()` в `dependencies.py`. Handler не змінюється.

### Складність реалізації та тестування

**Синхронний:** Простіший у реалізації та відлагодженні — один потік виконання, один call stack. Тести перевіряють прямий виклик: "після створення замовлення notification service отримав виклик".

**Асинхронний:** Потребує EventBus інфраструктури, підписки, ідемпотентності обробників. Тести складніші — треба перевіряти, що подія опублікована, що підписник її отримав, що при повторній доставці обробник не виконується двічі. Зате тести handler'а ізольовані від тестів підписника.

## Який підхід для production-системи?

Для SmartBook Inventory оптимальний **гібридний підхід:**

- **Синхронно:** Операції, результат яких потрібен одразу (перевірка оплати, якби вона була).
- **Асинхронно:** Нотифікації, аналітика, аудит — все, що може зачекати секунду.

В нашому випадку обидві побічні операції (restocking + confirmation) — асинхронні за природою. Email може зачекати. Запит постачальнику може зачекати. Тому `async` — кращий вибір за замовчуванням (налаштовується через `COMMUNICATION_MODE=async` в `.env`).

## Ідемпотентність обробників

`NotificationEventHandler` реалізує deduplication через in-memory set `_processed_events`. Кожна подія має унікальний ключ (`low_stock:{book_id}`, `order_confirm:{order_id}`). При повторній доставці тієї ж події обробник перевіряє ключ і пропускає обробку.

Для production in-memory deduplication недостатньо (втрачається при рестарті). Потрібна персистентна таблиця оброблених подій або idempotency keys в БД. Але для моноліту з in-process EventBus повторна доставка малоймовірна, тому поточна реалізація прийнятна.
